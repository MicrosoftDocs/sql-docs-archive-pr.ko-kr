## <a name="invoking-a-natively-compiled-stored-procedure-from-an-ssis-package"></a><span data-ttu-id="512c0-108">SSIS 패키지에서 고유하게 컴파일된 저장 프로시저 호출</span><span class="sxs-lookup"><span data-stu-id="512c0-108">Invoking a natively compiled stored procedure from an SSIS Package</span></span>  
 <span data-ttu-id="512c0-109">SSIS 패키지에서 고유하게 컴파일된 저장 프로시저를 호출하려면 ODBC 원본이나 ODBC 대상을 **EXEC** 키워드 없는 **\<procedure name>** 형식의 SQL 문으로 사용하는 것이 좋습니다.</span><span class="sxs-lookup"><span data-stu-id="512c0-109">To invoke a natively compiled stored procedure from an SSIS package, we recommend that you use an ODBC Source or ODBC Destination with an SQL statement of the format: **\<procedure name>** without the **EXEC** keyword.</span></span> <span data-ttu-id="512c0-110">SQL 문에 EXEC 키워드를 사용하면 ODBC 연결 관리자가 SQL 명령 텍스트를 저장 프로시저 대신 [!INCLUDE[tsql](../../includes/tsql-md.md)] 문으로 해석하고 고유하게 컴파일된 저장 프로시저의 실행에 지원되지 않는 커서를 사용하기 때문에 오류 메시지가 나타납니다.</span><span class="sxs-lookup"><span data-stu-id="512c0-110">If you use the EXEC keyword in the SQL statement, you will see an error message because the ODBC connection manager interprets the SQL command text as a [!INCLUDE[tsql](../../includes/tsql-md.md)] statement rather than a stored procedure and use cursors, which are not supported for execution of natively compiled stored procedures.</span></span> <span data-ttu-id="512c0-111">연결 관리자는 EXEC 키워드 없는 SQL 문을 저장 프로시저 호출로 처리하고 커서를 사용하지 않습니다.</span><span class="sxs-lookup"><span data-stu-id="512c0-111">The connection manager treats the SQL statement without the EXEC keyword as a stored procedure call and will not use a cursor.</span></span>  
  
 <span data-ttu-id="512c0-112">ADO .NET 원본과 OLE DB 원본을 사용하여 고유하게 컴파일된 저장 프로시저를 호출할 수도 있지만 ODBC 원본을 사용하는 것이 좋습니다.</span><span class="sxs-lookup"><span data-stu-id="512c0-112">You can also use ADO .NET Source and OLE DB Source to invoke a natively compiled stored procedure, but we recommend that you use ODBC Source.</span></span> <span data-ttu-id="512c0-113">고유하게 컴파일된 저장 프로시저를 실행하도록 ADO .NET 원본을 구성하는 경우, ADO .NET 원본이 기본적으로 사용하는 [!INCLUDE[ssNoVersion](../../includes/ssnoversion-md.md)] (SqlClient)의 데이터 공급자가 고유하게 컴파일된 저장 프로시저의 실행을 지원하지 않기 때문에 오류 메시지가 나타납니다.</span><span class="sxs-lookup"><span data-stu-id="512c0-113">If you configure the ADO .NET Source to execute a natively compiled stored procedure, you will see an error message because the data provider for [!INCLUDE[ssNoVersion](../../includes/ssnoversion-md.md)] (SqlClient), which the ADO .NET Source uses by default, does not support execution of natively compiled stored procedures.</span></span> <span data-ttu-id="512c0-114">ODBC 데이터 공급자, [!INCLUDE[ssNoVersion](../../includes/ssnoversion-md.md)]또는 [!INCLUDE[ssNoVersion](../../includes/ssnoversion-md.md)] Native Client를 사용하도록 ADO .NET 원본을 구성할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="512c0-114">You can configure the ADO .NET Source to use the ODBC Data Provider, OLE DB Provider for [!INCLUDE[ssNoVersion](../../includes/ssnoversion-md.md)], or [!INCLUDE[ssNoVersion](../../includes/ssnoversion-md.md)] Native Client.</span></span> <span data-ttu-id="512c0-115">하지만 ODBC 데이터 공급자와 함께 ADO .NET 원본을 사용하는 것보다 ODBC 원본이 더 좋은 성능을 나타냅니다.</span><span class="sxs-lookup"><span data-stu-id="512c0-115">However, note that the ODBC Source performs better than ADO .NET Source with ODBC Data Provider.</span></span>
